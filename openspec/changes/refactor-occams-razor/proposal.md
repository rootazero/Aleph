# Change Proposal: refactor-occams-razor

## Metadata
- **ID**: refactor-occams-razor
- **Title**: Occam's Razor Refactoring - Eliminate Unnecessary Code Complexity
- **Type**: Technical Debt / Refactoring
- **Status**: Draft
- **Created**: 2026-01-02

## Overview

### Problem Statement
The Aether codebase has accumulated **18 identified violations** of Occam's Razor principle ("Entities should not be multiplied without necessity"), affecting approximately **480 lines of code** across both Rust core and Swift UI layers:

1. **High Severity (6 violations)**: Core logic issues causing significant maintenance burden
   - Excessive mutex lock boilerplate (20+ repetitions in `core.rs`)
   - Redundant memory DB null checks (10+ methods)
   - Duplicated provider menu rebuild logic (2 near-identical methods)
   - Redundant error conversion boilerplate
   - Over-complex nested match statements (3-level deep async)
   - Unused dependency (`tokio-util`) adding build time bloat

2. **Medium Severity (8 violations)**: Architectural inefficiencies
   - Duplicated config reload observers (Rust + Swift overlap)
   - Over-abstracted `ProviderConfigEntry` wrapper
   - Redundant provider type inference
   - Excessive `.clone()` operations (198 occurrences, some redundant)
   - Duplicated test provider logic (90% code overlap)
   - Over-engineered error type hierarchy (13 variants)
   - Redundant permission check methods
   - Duplicated alert creation logic (3 repetitions)

3. **Low Severity (4 violations)**: Minor inefficiencies
   - Unused dependencies (`futures_util`, `once_cell`)
   - Redundant color parsing logic
   - Over-use of `.into()` conversions reducing readability

**Impact**: These violations increase cognitive load, reduce maintainability, and slow down build times.

### Proposed Solution

Implement a **Three-Phase Safety-First Refactoring** approach:

#### **Phase 1: The Detective (Scan & Tag)**
- **Complete**: Automated codebase analysis has identified all 18 violations
- **Output**: `STEP1_CANDIDATES.md` with detailed analysis (already generated by exploration agent)

#### **Phase 2: The Judge (Verify & Filter)**
- Apply risk assessment against **Critical Constraints**:
  - ✅ **UniFFI Integrity**: Never touch `#[uniffi::export]`, `Arc<T>` wrappers, generated bindings
  - ✅ **FFI Safety**: Preserve memory layout (`#[repr(C)]`), public signatures
  - ✅ **Logic Preservation**: Input/Output behavior must remain identical
  - ✅ **Generated Code**: Ignore all auto-generated UniFFI files
- Filter out false positives and high-risk changes
- **Output**: `STEP2_VERIFIED_PLAN.md` with safe, high-value tasks

#### **Phase 3: The Surgeon (Execute)**
- Apply changes ONE BY ONE with surgical precision
- Validate against constraints after each change
- Abort and revert if unexpected complexity emerges

## Impact Analysis

### User Experience
- **Neutral**: No visible changes to end users (internal refactoring only)
- **Indirect Benefit**: Faster build times, more stable codebase

### Technical Complexity
- **High Risk Items** (will be excluded in Phase 2):
  - Any changes to UniFFI interface definitions
  - FFI boundary modifications
  - Complex async/concurrency patterns

- **Low Risk Items** (high priority):
  - Helper method extraction (mutex locks, null checks)
  - Dependency removal (unused crates)
  - Code duplication elimination (menu builders, alerts)

### Performance Impact
- **Build Time**: Expected 5-10% reduction via dependency removal
- **Runtime**: Neutral (or minor improvements from reduced clones)
- **Binary Size**: ~2-5% reduction via dependency cleanup

### Dependencies
This refactoring affects multiple specs but maintains compatibility:
- `core-library` (MODIFIED): Core logic simplification
- `uniffi-bridge` (UNCHANGED): No interface changes
- `macos-client` (MODIFIED): UI helper method extraction
- `ai-provider-interface` (MODIFIED): Test method consolidation

## Alternatives Considered

### Alternative 1: "Big Bang" Refactoring
- Refactor all 18 violations simultaneously
- **Rejected**: Too risky, difficult to isolate issues, violates incremental approach

### Alternative 2: No Refactoring (Accept Technical Debt)
- Leave code as-is, focus on new features
- **Rejected**: Technical debt compounds over time, maintenance burden grows

### Alternative 3: Automated Refactoring Tools
- Use `rust-analyzer` auto-refactors, Swift automated refactoring
- **Rejected**: Tools don't understand UniFFI constraints, require manual review anyway

## Open Questions

1. **Should we refactor error type hierarchy?**
   - **Proposed**: YES (Phase 3), consolidate 13 variants → 6-8 broader categories
   - **Risk**: MEDIUM (affects error handling throughout codebase)

2. **Should we remove `ProviderConfigEntry` wrapper?**
   - **Proposed**: INVESTIGATE (Phase 2), check if UniFFI can serialize `HashMap<String, ProviderConfig>` directly
   - **Risk**: HIGH (UniFFI interface change, may require Swift updates)

3. **What's the threshold for "acceptable" `.clone()` usage?**
   - **Proposed**: Keep clones at FFI boundaries, eliminate internal redundant clones
   - **Risk**: LOW (local code changes only)

4. **Should we consolidate error variants before or after other refactors?**
   - **Proposed**: AFTER (Phase 3 final step), as it touches many files
   - **Risk**: MEDIUM (large surface area)
