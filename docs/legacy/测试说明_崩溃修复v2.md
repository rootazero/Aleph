# 🔧 关键崩溃修复完成 (第二次修复)

## ✅ 问题已解决

**问题**：在备忘录或其他应用输入文字时 Aleph 闪退
**错误**：`_dispatch_assert_queue_fail` in `TSMGetInputSourceProperty`
**根本原因**：rdev 在后台线程调用 macOS 输入法 API（需要主线程）

---

## 🎯 最终解决方案

### 第一次尝试（失败）
- **方法**：升级 rdev 从 0.5.x 到 0.6.0 (git main)
- **结果**：❌ 崩溃依然发生
- **原因**：rdev 0.6.0 有主线程支持，但默认 `is_main_thread = true`

### 第二次修复（成功）✅
- **方法**：调用 `rdev::set_is_main_thread(false)` 告诉 rdev 使用主队列
- **原理**：rdev 内部会使用 `dispatch::Queue::main().exec_sync()` 来执行 `TSMGetInputSourceProperty`
- **代码位置**：`Aether/core/src/hotkey/rdev_listener.rs:86`

### 关键代码
```rust
#[cfg(target_os = "macos")]
{
    // CRITICAL FIX: Tell rdev we are NOT on main thread
    // This prevents _dispatch_assert_queue_fail crash
    rdev::set_is_main_thread(false);
    eprintln!("[RdevListener] Set is_main_thread=false for proper macOS main thread handling");
}
```

### rdev 内部机制
```rust
// rdev/src/macos/keyboard.rs:95-102
if self.is_main_thread {
    self.string_from_code(code, modifier_state)
} else {
    QUEUE.exec_sync(move || {  // ✅ 使用主队列
        self.string_from_code(code, modifier_state)
    })
}
```

## 🔍 技术细节

### 为什么第一次修复失败？
虽然 rdev 0.6.0 有主线程支持，但需要**显式设置** `is_main_thread = false`。

### 为什么这次修复会成功？
```
调用链：
rdev::listen() (后台线程)
    ↓
keyboard::create_string_for_key()
    ↓
检查 is_main_thread == false ✅
    ↓
使用 QUEUE.exec_sync()  ← 关键！
    ↓
在主队列执行 string_from_code()
    ↓
TSMGetInputSourceProperty (主队列 ✅)
    ↓
无崩溃！
```

### Dispatch Queue 断言
```c
// macOS 系统断言
void _dispatch_assert_queue_fail(dispatch_queue_t queue) {
    // 检测到 TSMGetInputSourceProperty 在非主队列调用
    // 触发 SIGTRAP → 应用终止
}
```

### rdev 修复对比
```rust
// ❌ 第一次（失败）
// 仅升级 rdev，但 is_main_thread = true（默认值）
rdev::listen() → 后台线程 → 直接调用 → 崩溃

// ✅ 第二次（成功）
// 设置 is_main_thread = false
rdev::set_is_main_thread(false);
rdev::listen() → 后台线程 → 主队列 exec_sync → 成功
```
