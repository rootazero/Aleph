# Memory VFS Evolution Design

> From "Flat Fact Store" to "Hierarchical Knowledge Library"

**Date**: 2026-02-22
**Status**: Approved
**Reference**: OpenViking (https://github.com/volcengine/OpenViking)

---

## 1. Motivation

Aleph's current memory system is a **database-centric** architecture storing atomized "Facts" in SQLite + sqlite-vec. While effective for discrete information retrieval, it lacks hierarchy and global perspective when handling complex, structured, long-form context.

### Core Problem: Fragmentation

When an Agent needs to understand a complex concept (e.g., "project architecture"), it must retrieve and reassemble dozens of individual Facts. This leads to:

- **Information loss**: Key connections between Facts are invisible
- **Token waste**: No "zoom level" — either load everything or nothing
- **No spatial awareness**: "Python preferences" and "Project A API key" are just two records with different IDs, no structural separation

### Target State: Cognitive Management

Evolve from pure "retrieval" to "cognitive management" — the Agent should know **what it knows**, **where to find it**, and **how deep to go**.

---

## 2. Approach: Progressive Hybrid Architecture

**Preserve existing SQLite infrastructure** while adding a navigation layer on top. We are not replacing the fact store — we are adding a "library catalog" to the "sticky note box".

### Why Not a Full Rewrite?

| Existing Asset | Value |
|---|---|
| `VectorDatabase` (sqlite-vec) | High-performance local vector search |
| `CompressionDaemon` | Idle-time compression with LLM extraction |
| `ValueEstimator` | 8-signal importance scoring |
| `HybridRetrieval` | Vector + BM25 RRF fusion |
| `ContextComptroller` | Post-retrieval deduplication |
| `EvolutionChain` | Contradiction detection and resolution |

All these remain untouched. We add structure on top.

### 3.4 Path Namespace Design

```
aleph://
├── user/                   # About the "master" (user)
│   ├── preferences/        # FactType::Preference
│   ├── personal/           # FactType::Personal
│   └── plans/              # FactType::Plan
├── knowledge/              # About "things" (domain knowledge)
│   ├── learning/           # FactType::Learning
│   └── projects/           # FactType::Project
├── agent/                  # About "me" (the agent)
│   ├── tools/              # FactType::Tool
│   └── experiences/        # Cortex experiences
└── session/                # Temporary session data
    └── {session_id}/
```

**Design Rationale**:
- `user/` stores information about the master — preferences, identity, plans
- `knowledge/` stores domain knowledge — learning records, project docs
- `agent/` stores the agent's own knowledge — tool experiences, execution patterns
- `session/` stores ephemeral session context

**Shadow Paths**: The same project can appear in both:
- `aleph://knowledge/projects/aleph/` (reference documentation)
- `aleph://agent/experiences/aleph/` (agent's hands-on lessons learned)

## 4. L1 Overview Generation

### 4.1 L1 Overview Characteristics

L1 Overviews are stored as `FactSource::Summary` Facts:

| Attribute | Value |
|---|---|
| `fact_source` | `Summary` |
| `path` | The directory path itself, e.g., `aleph://user/preferences/` |
| `content` | Structured Markdown summary (~500-1000 tokens) |
| `confidence` | 1.0 (system-generated) |
| `temporal_scope` | `Permanent` |
| `specificity` | `Principle` (highest level) |
| `content_hash` | Hash of child Facts for staleness detection |

### 4.2 Staleness Detection

```rust
/// Compute content hash for all L2 Facts under a path
fn compute_directory_hash(facts: &[MemoryFact]) -> String {
    let mut hasher = Sha256::new();
    let mut sorted: Vec<_> = facts.iter().collect();
    sorted.sort_by_key(|f| &f.id);
    for fact in sorted {
        hasher.update(fact.id.as_bytes());
        hasher.update(fact.updated_at.to_be_bytes());
    }
    hex::encode(hasher.finalize())[..16].to_string()
}
```

### 4.3 Generation Trigger: CompressionDaemon Extension

```
Existing flow:
  Idle → Check uncompressed conversations → LLM extract Facts → Store in DB

New step (after Fact extraction):
  → Collect affected paths (set of paths from new Facts)
  → For each affected path:
     → Compute content_hash from child L2 Facts
     → Compare with existing L1 Summary Fact's content_hash
     → If stale → Regenerate L1 Overview via LLM
     → Store/update L1 Summary Fact
  → Bubble up: If child L1 changed, mark parent path as needing update
```

### 4.4 L1 Generation Prompt

```
You are generating an L1 Overview for the knowledge directory: {path}

Below are the L2 facts stored under this path:
{facts_list}

Generate a structured Markdown overview that:
1. Summarizes the key themes (3-5 bullet points)
2. Lists each sub-topic with a one-line description
3. Notes any contradictions or evolution
4. Total length: 500-1000 tokens

Format:
# {path_display_name}

## Key Themes
- ...

## Contents
- **{sub_topic_1}**: {one_line_description}
- **{sub_topic_2}**: {one_line_description}

## Notes
- {any_contradictions_or_evolution}
```

### 4.5 Agent Bootstrapping

At Agent startup, top-level L1 Overviews are loaded as part of the System Prompt:

```rust
async fn bootstrap_agent_context(&self) -> Vec<MemoryFact> {
    let top_level_paths = [
        "aleph://user/",
        "aleph://knowledge/",
        "aleph://agent/",
    ];

    let mut overviews = Vec::new();
    for path in top_level_paths {
        if let Some(l1) = self.find_l1_overview(path).await {
            overviews.push(l1);
        }
    }
    overviews
}
```

This gives the Agent immediate self-awareness: "Here's what I know about this user."

## 5. Tool System Changes

### 5.1 memory_search Enhancement

The existing `memory_search` output gains path-based clustering:

```rust
pub struct MemorySearchOutput {
    // Existing fields
    pub facts: Vec<FactResult>,
    pub transcripts: Vec<TranscriptResult>,
    pub query: String,
    pub tokens_saved: usize,

    // New: path-clustered results
    pub path_clusters: Vec<PathCluster>,
}

pub struct PathCluster {
    pub path: String,                  // "aleph://user/preferences/coding/"
    pub l1_overview: Option<String>,   // L1 summary content (if available)
    pub fact_count: usize,             // Number of Facts hit under this path
    pub top_score: f32,                // Highest similarity score
}
```

**Retrieval logic change**:
1. Execute existing Vector + BM25 hybrid retrieval
2. **New**: Group results by `path`
3. If a path has >= 3 hit Facts → return its L1 Overview + PathCluster instead
4. If a path has < 3 hit Facts → return Facts directly (existing behavior)

### 5.2 New Tool: memory_browse

### 5.3 Zoom Retrieval Interaction Flow

```
Step 1: Agent calls memory_search(query="Python coding preferences")

Step 2: memory_search returns:
  {
    "path_clusters": [
      {
        "path": "aleph://user/preferences/coding/",
        "l1_overview": "User's Python/Rust coding preferences including PEP8, naming conventions, and commonly used libraries.",
        "fact_count": 8,
        "top_score": 0.85
      }
    ],
    "facts": [ ... individual low-cluster facts ... ]
  }

Step 3: Agent decides to drill down:
  memory_browse(action='ls', path='aleph://user/preferences/coding/')

Step 4: Agent reads specific detail:
  memory_browse(action='read', path='aleph://user/preferences/coding/style.md')
```

### 5.4 Tool Separation Rationale

| Tool | Focus | Analogy |
|---|---|---|
| `memory_search` | **What** to find (semantic) | Google Search |
| `memory_browse` | **Where** to look (structural) | File Explorer |

This separation keeps each tool's API clean and gives the Agent **metacognitive ability** — it knows what it has, roughly what it contains, and how to get more detail.

## 6. Markdown Document Facts

### 6.1 Long Document Storage

Support storing complete Markdown documents without forced chunking:

```rust
pub struct DocumentFact {
    pub path: String,           // "aleph://knowledge/projects/aleph/architecture.md"
    pub content: String,        // Full Markdown content
    pub fact_source: FactSource::Document,
    pub l0_abstract: String,    // Auto-generated <100 token summary (for vector index)
}
```

### 6.2 Storage Strategy

1. Original Markdown stored in `content` field
2. Auto-generate L0 Abstract, embed as vector for indexing
3. If document exceeds ~4000 tokens → auto-chunk by semantic boundaries, each chunk becomes an independent Fact sharing the same `parent_path`

## 7. Data Migration

### 7.1 Existing Facts Migration

**Step 1: Automated path assignment (based on FactType)**

```rust
async fn migrate_existing_facts(db: &VectorDatabase) {
    let facts = db.get_all_valid_facts().await;
    for fact in facts {
        let path = fact.fact_type.default_path();
        db.update_fact_path(&fact.id, path).await;
    }
}
```

**Step 2: Initial L1 generation**

After migration, trigger a full L1 generation pass as a special CompressionDaemon task.

### 7.2 Backward Compatibility

| Scenario | Behavior |
|---|---|
| Fact with empty `path` | Retrieval works normally (existing logic doesn't depend on path) |
| Path without L1 Overview | `memory_search` returns Facts directly, no clustering |
| `memory_browse` on empty path | Returns empty result + hint "No content at this path" |
| Old Client version | Ignores `path_clusters` field in response |

**Principle**: All new features degrade gracefully to existing behavior.

## 10. Implementation Phases

### Phase 1: Path Infrastructure
- Add `path`, `fact_source`, `content_hash`, `parent_path` to schema
- Implement `FactType::default_path()` mapping
- Migrate existing Facts
- Add path prefix index and query support

### Phase 2: L1 Overview Generation
- Implement `compute_directory_hash()`
- Extend CompressionDaemon with L1 generation step
- Design and test L1 generation prompt
- Implement staleness detection and re-generation

### Phase 3: Tool System
- Enhance `memory_search` output with `PathCluster`
- Implement `memory_browse` tool (ls/read/glob)
- Add Agent bootstrapping (load top-level L1s on startup)
- Add navigation hints in search results

### Phase 4: Document Facts
- Support `FactSource::Document` storage
- Auto-generate L0 abstracts for long documents
- Implement semantic chunking for oversized documents

---

## 11. Key Design Principles

1. **Graceful degradation**: Every new feature falls back to existing behavior
2. **No storage engine change**: SQLite remains the single source of truth
3. **Reuse over rewrite**: CompressionDaemon, ValueEstimator, HybridRetrieval all stay
4. **Agent autonomy**: The Agent decides when to zoom in, not the system
5. **Path as index**: Prefix search enables fast `ls` operations
6. **Compression-time generation**: L1 Overviews are generated during idle time, not at write time
