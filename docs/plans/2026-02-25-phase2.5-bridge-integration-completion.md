# Phase 2.5: Bridge Integration Completion

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Close the gap between Phase 1's IPC skeleton and Phase 3's desktop capabilities by parsing handshake capabilities, making WebView URLs dynamic, and unifying the two bridge client paths.

**Architecture:** Phase 1 created two parallel bridge access paths — `DesktopBridgeClient` (stateless, per-request UDS at `~/.aleph/bridge.sock`) and `DesktopBridgeManager` (persistent transport at `~/.aleph/run/desktop-bridge.sock` via supervisor). These point at *different sockets*, which is a critical inconsistency. Phase 2.5 unifies them: the manager becomes the single source of truth for bridge access, the client delegates to the manager when available, and capabilities from handshake drive tool availability.

**Tech Stack:** Rust (tokio, serde_json), Tauri v2, existing `BridgeSupervisor` + `Transport` traits

**Step 2: Navigate windows to correct URL in setup**

```rust
    .ok()
    .and_then(|s| s.parse().ok())
    .unwrap_or(18790);

    let url = format!("{}/halo", base_url);
    if let Ok(parsed) = url.parse() {

    }

    let url = format!("{}/settings", base_url);
    if let Ok(parsed) = url.parse() {
        let _ = settings.navigate(parsed);
    }
}

tracing::info!(port = server_port, "WebView windows configured for server at {}", base_url);
```

### Task 4: Unify DesktopBridgeClient Socket Path

**Files:**
- Modify: `core/src/desktop/client.rs`

**Context:** There are two socket paths in play:
- `DesktopBridgeClient` connects to `~/.aleph/bridge.sock` (the "standalone" socket)
- `BridgeSupervisor` creates `~/.aleph/run/desktop-bridge.sock` (the "managed" socket)

When the bridge runs in standalone mode (user launches it manually), it listens on `~/.aleph/bridge.sock`. When launched by the supervisor, it listens on `~/.aleph/run/desktop-bridge.sock` (from `ALEPH_SOCKET_PATH` env var).

`DesktopBridgeClient` needs to try both paths: first the managed socket (if the server started the bridge), then fallback to the standalone socket.

**Step 1: Update `DesktopBridgeClient::new()` to try managed socket first**

```rust
impl DesktopBridgeClient {
    /// Create a new client.
    ///
    /// Probes two socket paths in order:
    /// 1. `~/.aleph/run/desktop-bridge.sock` (managed by BridgeSupervisor)
    /// 2. `~/.aleph/bridge.sock` (standalone bridge)
    pub fn new() -> Self {
        let home = dirs::home_dir().expect("cannot resolve home directory");
        let managed = home.join(".aleph").join("run").join("desktop-bridge.sock");
        let standalone = home.join(".aleph").join("bridge.sock");

        // Prefer managed socket (from BridgeSupervisor) when it exists
        let socket_path = if managed.exists() { managed } else { standalone };
        Self { socket_path }
    }
}
```

**Step 2: Add `with_path()` constructor for tests**

```rust
/// Create a client pointing at a specific socket path (for testing).
pub fn with_path(socket_path: PathBuf) -> Self {
    Self { socket_path }
}
```

**Step 3: Update doc comment on module**

Update the module-level comment in `client.rs`:

```rust
//! Unix Domain Socket client for the Desktop Bridge.
//!
//! Uses JSON-RPC 2.0 over a newline-delimited stream. Each request opens a
//! fresh connection, sends one JSON-RPC request, reads one response, and closes.
//!
//! Socket path precedence:
//! 1. `~/.aleph/run/desktop-bridge.sock` (bridge launched by BridgeSupervisor)
//! 2. `~/.aleph/bridge.sock` (bridge launched standalone)
```
