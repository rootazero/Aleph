# Markdown Tool Adapter - Design Document

**Date:** 2026-02-04
**Status:** Approved for Implementation
**Priority:** Phase 1 of Evolution Loop Architecture

---

## Executive Summary

Implement a **Markdown Tool Adapter** to enable Aleph to dynamically load CLI tools defined in Markdown files (SKILL.md format), compatible with OpenClaw ecosystem while maintaining Aleph's type safety and security model.

### Strategic Context

This is not just ecosystem compatibility—it's the **missing link** for Aleph's Evolution Loop:
- **Problem**: Aleph's current tools are Rust code (compiled, static)
- **Vision**: Evolution system should generate executable skills at runtime
- **Solution**: Markdown format provides a "hot-pluggable, text-defined, runtime-loadable" skill format

Without this adapter, the Evolution Loop would need to generate Rust code and recompile itself—practically impossible. With it, discovered patterns can be written as Markdown skills and loaded immediately.

---

## Design Goals

1. **OpenClaw Compatibility**: Zero-modification loading of existing OpenClaw skills
2. **Aleph Enhancement**: Extend format with type hints and security controls
3. **Safety First**: Maintain Aleph's security model (sandbox, confirmation, network isolation)
4. **Evolution Ready**: Enable future auto-generation of skills from usage patterns

## Specification: Aleph SKILL.md Format

### Base Format (OpenClaw Compatible)

```markdown
---
name: github-pr
description: "Manage GitHub Pull Requests"
metadata:
  requires:
    bins: ["gh"]
---

# GitHub PR Tool

Use `gh pr list` to list pull requests.

## Examples

\`\`\`bash
gh pr list --repo owner/name
gh pr view 123
\`\`\`
```

### Aleph Extensions (Optional)

```markdown
---
name: github-pr
description: "Manage GitHub Pull Requests"
metadata:
  requires:
    bins: ["gh"]

  # Aleph-specific extensions
  aether:
    # Security controls
    security:
      sandbox: "host"           # 'host' | 'docker' | 'virtual_fs'
      confirmation: "write"     # 'always' | 'write' | 'never'
      network: "internet"       # 'internet' | 'local' | 'none'

    # Type hints for better JSON Schema generation
    input_hints:
      repo:
        type: "string"
        pattern: "^[^/]+/[^/]+$"
        description: "Repository in format owner/name"
        optional: false
      number:
        type: "integer"
        description: "PR number"
        optional: true

    # Docker configuration (required if sandbox: docker)
    docker:
      image: "ghcr.io/cli/cli:2.40.0"
      env_vars:
        - GITHUB_TOKEN
        - GH_TOKEN
      extra_flags:
        - "-v"
        - "${HOME}/.gitconfig:/root/.gitconfig:ro"

    # Evolution metadata (auto-generated by system)
    evolution:
      source: "auto-generated"
      confidence_score: 0.95
      created_from_trace: "trace-id-12345"
---

# Content...
```

## Data Structures

### Core Spec (spec.rs)

```rust
pub struct AlephSkillSpec {
    pub name: String,
    pub description: String,
    pub metadata: SkillMetadata,
    pub markdown_content: String,
}

pub struct SkillMetadata {
    pub requires: RequiresSpec,
    pub aether: Option<AetherExtensions>,
}

pub struct AlephExtensions {
    pub security: SecuritySpec,
    pub input_hints: HashMap<String, InputHint>,
    pub evolution: Option<EvolutionMeta>,

pub struct SecuritySpec {
    pub sandbox: SandboxMode,      // Host | Docker | VirtualFs
    pub confirmation: ConfirmationMode,  // Always | Write | Never

pub struct InputHint {
    pub hint_type: Option<String>,
    pub pattern: Option<String>,
    pub values: Option<Vec<String>>,
    pub description: Option<String>,
    pub optional: bool,
}

pub struct DockerConfig {
    pub image: String,
    pub env_vars: Vec<String>,
    pub extra_flags: Vec<String>,
}
```

## Key Design Decisions

### 1. JSON Schema Generation Strategy

**Problem**: Markdown descriptions are rich but unstructured. JSON Schema needs structure.

**Solution**: Hybrid approach
- **With input_hints**: Generate typed schema with validation (pattern, enum, etc.)
- **Without hints**: Fallback to `args: array<string>` for safety
- **Always include**: `required` fields to prevent LLM hallucination

**Critical**: Fallback uses `args` array (not single `command` string) to prevent Shell Injection.

```rust
// Example generated schema (with hints)
{
  "type": "object",
  "properties": {
    "repo": {"type": "string", "pattern": "^[^/]+/[^/]+$"},
    "number": {"type": "integer"}
  },
  "required": ["repo"],
  "additionalProperties": false
}

// Fallback schema (no hints)
{
  "type": "object",
  "properties": {
    "args": {
      "type": "array",
      "items": {"type": "string"},
      "description": "Command-line arguments as separate strings"
    }
  },
  "required": ["args"],
  "additionalProperties": false
}
```

### 2. Context Injection Strategy

**Problem**: LLM needs usage examples, but token budget is limited.

**Solution**: Mixed strategy (Smart Tool Discovery compatible)
1. **Definition Phase**: Extract `## Examples` section, attach to `ToolDefinition.llm_context`
2. **Prompt Building**: PromptBuilder injects `llm_context` into system prompt alongside schema
3. **No Runtime Injection**: Context is static, not injected during `call()`

**Architecture Change Required**: Extend `ToolDefinition` with optional `llm_context` field.

### 3. CLI Argument Conversion

**Problem**: Converting JSON to CLI args is fragile (positional args, subcommands, flag styles vary).

**Solution**: Prioritize safety over cleverness
- **Primary Mode**: `args` array (safest, direct pass-through)
- **Fallback Mode**: Typed object → `--flag value` conversion
  - Only for simple flag-based tools
  - Preserves order via `input_hints` or alphabetical
  - Logs warning about potential fragility

**Guidance**: Encourage LLM to use `args` array for complex tools.

### 4. Execution Isolation

**Host Mode**:
- Direct `tokio::Command` execution
- Optional network restriction via env vars
- Binary availability checked at load time (via `which`)

**Docker Mode**:
- Strict image resolution (no blind fallback to alpine)
- Environment variable pass-through via `docker.env_vars` allowlist
- Enhanced exit code handling (125/126/127/137)
- Security hardening: `--read-only`, `--tmpfs /tmp`, `noexec`

**Critical Fix**: Only check binary availability for `sandbox: host`. Docker mode runs binaries inside container.

### 5. Error Handling Philosophy

**Partial Failure Tolerance**: One bad skill should not break the entire system.

```rust
pub async fn load_all(&self) -> (Vec<MarkdownCliTool>, Vec<(PathBuf, Error)>)
```

- Returns both successes and failures
- Logs warnings for failures
- Allows system to start with partial skill library

## Security Considerations

1. **Shell Injection Prevention**:
   - Use `args` array (not string concatenation)
   - Never use `sh -c` wrapping

2. **Docker Isolation**:
   - Read-only root filesystem
   - Limited tmpfs with `noexec`
   - Network mode restrictions
   - No automatic `--privileged`

3. **Environment Variable Leakage**:
   - Only pass explicitly allowlisted env vars
   - No automatic `--env-file` mounting

4. **Binary Validation**:
   - Check existence at load time (host mode only)
   - Strict image resolution (fail if unknown)

## Open Questions / Technical Debt

1. **Positional Arguments**: Current `args_to_cli` doesn't handle subcommands elegantly. Consider adding `arg_format: positional | flag | subcommand` to `InputHint`.

2. **Docker Image Discovery**: Hardcoded mapping is brittle. Consider:
   - Online registry lookup (docker hub API)
   - Local cache of known images
   - User-provided mapping config

3. **Async Recursion vs Walkdir**: Current design uses `walkdir` in `spawn_blocking`. Consider fully async solution if performance becomes issue.

```markdown
---
```

```markdown
  requires:
    bins: ["kubectl"]
  aether:

      sandbox: docker
      confirmation: never
```
