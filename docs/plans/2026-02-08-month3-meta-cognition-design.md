## Executive Summary

Month 3 introduces the **Meta-Cognition Layer** to Aleph, enabling the system to observe, critique, and improve its own thinking process. This transforms Aleph from a "learning executor" into a "self-reflective intelligence" that can identify failure patterns, optimize behavior, and evolve its decision-making strategies.

**Core Philosophy**: Combine "pain learning" (immediate response to failures) with "sleep consolidation" (periodic optimization during idle time), mirroring biological intelligence evolution.

## Design Principles

### 1. Hybrid Reflection Mode

**Reactive Reflection (Pain Learning)**
- Triggered by explicit failure signals
- Immediate response to errors, negative feedback, intent drift
- High priority (100), high initial confidence (0.8)
- Generates corrective behavioral anchors

**Proactive Reflection (Excellence Learning)**
- Triggered during system idle time
- Scans for "successful but mediocre" tasks
- Medium priority (50), lower initial confidence (0.6)
- Generates optimization behavioral anchors

### 2. Constitution vs. Case Law

**Constitution Layer (Immutable)**
- Core system prompt defining identity, ethics, base logic
- Written by developers, version-controlled
- Never modified by meta-cognition layer

**Case Law Layer (Evolvable)**
- `learned_rules.md` containing behavioral anchors
- Generated by meta-cognition from experience
- Tag-based, dynamically injected when relevant
- Can be reviewed, rolled back, deprecated

### 3. Dynamic Anchor Injection

Rules are not blindly loaded into every request. Instead:
1. Extract tags from user intent and execution context
2. Retrieve relevant anchors from store (tag match + vector similarity)
3. Inject only 3-5 most relevant rules into system prompt
4. Maintain token efficiency while preserving wisdom

## Core Components

### 1. BehavioralAnchor Data Structure

```rust
pub struct BehavioralAnchor {
    pub id: String,                    // UUID
    pub rule_text: String,             // Human-readable instruction
    pub trigger_tags: Vec<String>,     // ["Python", "macOS", "shell"]
    pub confidence: f32,               // 0.0-1.0, increases with validation
    pub created_at: DateTime<Utc>,
    pub last_validated: DateTime<Utc>,
    pub validation_count: u32,         // How many times it helped
    pub failure_count: u32,            // How many times it was wrong
    pub source: AnchorSource,          // Where did this rule come from?
    pub scope: AnchorScope,            // When should it apply?
    pub priority: i32,                 // Higher = more important
    pub conflicts_with: Vec<String>,   // IDs of conflicting rules
    pub supersedes: Option<String>,    // ID of rule this replaces
}

pub enum AnchorSource {
    ReactiveReflection { task_id: String, error_type: String },
    ProactiveReflection { pattern_hash: String, optimization_type: String },
    UserFeedback { session_id: String },
    ManualInjection { author: String },
}

pub enum AnchorScope {
    Global,                           // Always apply
    Tagged { tags: Vec<String> },     // Apply when tags match
    Conditional { predicate: String }, // Apply when condition met
}
```

**Key Features:**
- **Confidence scoring**: Rules evolve based on validation/failure counts
- **Conflict tracking**: Explicit relationships between rules
- **Supersession chain**: New rules can replace old ones with history
- **Source tracking**: Know where each rule came from for debugging

### 2. Conflict Detection System

**Three-Phase Detection:**

**Phase 1: Semantic Similarity**
- Embed new rule and existing rules
- Calculate cosine similarity
- If similarity > 0.85: Redundant (keep higher confidence)
- If similarity > 0.70: Needs LLM review

**Phase 2: Logical Contradiction**
- Use LLM to detect logical conflicts
- Prompt: "Do these rules contradict each other?"
- Returns: `{contradicts: bool, reason: string}`

**Phase 3: Empirical Validation**
- Track real-world performance
- If task fails when rules A+B both apply
- But succeeds when only A applies
- Mark A and B as empirically conflicting

**Resolution Strategy:**
- Redundant: Keep higher confidence, mark other as superseded
- Contradictory: Present to user OR auto-resolve by confidence
- Empirically conflicting: Disable lower confidence, flag for review

### 3. Reactive Reflector

**Failure Signals:**

```rust
pub enum FailureSignal {
    ExecutionError {
        task_id: String,
        error: AlephError,
        context: ExecutionContext,
    },

    NegativeFeedback {
        session_id: String,
        user_message: String,
        previous_response: String,
    },

    ManifestValidationFailed {
        task_id: String,
        manifest: SuccessManifest,
        actual_result: TaskResult,
    },

    IntentDrift {
        task_id: String,
        original_intent: String,
        followup_question: String,
        time_gap_seconds: u64,
    },

    RepeatedFailure {
        pattern_hash: String,
        failure_count: u32,
        recent_attempts: Vec<String>,
    },
}
```

**Processing Flow:**
1. Receive failure signal
2. Create failure snapshot (intent, trace, error, context)
3. Analyze root cause using LLM
4. Generate corrective behavioral anchor
5. Set priority=100, confidence=0.8
6. Add to anchor store (with conflict detection)
7. Mark experience for priority distillation
8. If recoverable, suggest immediate retry

**Intent Drift Detection:**
- Monitor user follow-up questions within 60 seconds
- Even if Success Manifest passed
- Indicates incomplete understanding or execution
- Triggers reflection to improve future similar tasks

### 4. Critic Agent (Proactive Reflection)

**Trigger Conditions:**
- System idle for N seconds (configurable)
- OR scheduled task (e.g., 2 AM daily)
- Integrated with CortexDreamingService

**Scanning Process:**
1. Query experiences with status=Succeeded, reward 0.5-0.8
2. Analyze task chains for inefficiencies:
   - Redundant file reads
   - Unnecessary steps
   - Missing optimizations
3. Calculate efficiency score
4. If score < threshold, generate critic report

**Efficiency Metrics:**
- Redundant operations count
- Task chain length vs. optimal path
- Tool call efficiency
- Context switching overhead

**Report Generation:**
- Use LLM to analyze inefficient task chain
- Generate suggested behavioral anchor
- Set priority=50, confidence=0.6
- Requires more validation before high confidence

### 5. Anchor Store

**Storage:**
- SQLite table: `behavioral_anchors`
- Columns: id, rule_text, trigger_tags (JSON), confidence, priority, embedding (JSON), created_at, validation_count, failure_count, conflicts_with (JSON), supersedes
- Vector embeddings for semantic search

**Retrieval Strategy:**
1. Extract tags from intent (heuristic rules + LLM fallback)
2. Check LRU cache for tag-based results
3. If cache miss, query database:
   - Tag match (exact)
   - Vector similarity (semantic)
4. Rank by: confidence DESC, priority DESC
5. Return top 3-5 most relevant anchors
6. Update cache

**Cache Strategy:**
- LRU cache with configurable size (default: 100 entries)
- Key: tag string
- Value: Vec<BehavioralAnchor>
- Invalidate on anchor add/update/delete

### 6. Dynamic Injection

**Integration Point:**
- `agent_loop/message_builder/mod.rs`
- `build_system_prompt()` method

**Injection Process:**
1. Load immutable core system prompt
2. Retrieve relevant anchors for current intent
3. Format anchors as markdown section:
   ```
   ## Learned Behavioral Anchors
   Based on past experiences, follow these guidelines:

   - [Python, macOS] Always verify python3 path before execution
   - [Git, Conflict] Use strategy X when encountering merge conflicts
   ```
4. Append to core prompt
5. Log injected anchors for observability

**Token Optimization:**
- Only inject 3-5 most relevant rules
- Typical overhead: 100-200 tokens
- Negligible compared to context window (200K)

## Data Flow

### Reactive Flow (Failure → Learning)

```
Task Failure
    ↓
FailureSignal
    ↓
ReactiveReflector.handle_failure()
    ↓
Create FailureSnapshot
    ↓
Analyze Root Cause (LLM)
    ↓
Generate BehavioralAnchor (priority=100, confidence=0.8)
    ↓
Conflict Detection
    ↓
AnchorStore.add()
    ↓
Mark for Priority Distillation
    ↓
(Optional) Retry with Anchor
```

### Proactive Flow (Idle → Optimization)

```
System Idle
    ↓
CriticAgent.scan_for_improvements()
    ↓
Query Low-Efficiency Experiences
    ↓
Analyze Task Chains
    ↓
Generate CriticReport (LLM)
    ↓
Generate BehavioralAnchor (priority=50, confidence=0.6)
    ↓
Conflict Detection
    ↓
AnchorStore.add()
```

### Execution Flow (Task → Anchor Injection)

```
New Task Arrives
    ↓
Extract Intent
    ↓
AnchorStore.retrieve_for_intent()
    ↓
Tag Extraction (heuristic + LLM)
    ↓
Cache Check
    ↓
Database Query (tag match + vector similarity)
    ↓
Rank by Confidence & Priority
    ↓
MessageBuilder.build_system_prompt()
    ↓
Inject Top 3-5 Anchors
    ↓
Execute Task
    ↓
Record Outcome (validation_count++ or failure_count++)
```

## Evolution Path: Soft Rule → Hard Skill

The meta-cognition layer is the foundation for future "self-coding" capability:

1. **Stage 1 (Soft Rule)**: Text-based behavioral anchor injected into LLM context
2. **Stage 2 (Experience)**: Rule validated across multiple tasks, stored in experience replay buffer
3. **Stage 3 (Hard Skill)**: When rule is stable and high-frequency, system generates code:
   - "Instead of reminding LLM to check environment every time"
   - "Generate an `env_check` tool function"
   - Transition from text → executable code

This creates a natural progression from learned behavior to automated capability.

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Rule conflicts cause instability | High | Three-phase conflict detection, confidence-based resolution |
| LLM generates poor quality rules | Medium | Confidence scoring, empirical validation, manual review |
| Token overhead too high | Medium | Tag-based filtering, LRU cache, top-K selection |
| Prompt drift over time | High | Immutable constitution layer, version control for rules |
| False positive intent drift | Low | Configurable time threshold, semantic similarity check |
