# LLM Data Format Redesign

> Design document for enhancing LLM data transmission format to improve context retention and tool selection accuracy.

## Overview

### Problem Statement

After multiple iterations of intent detection, 3-layer routing, built-in runtimes, rig-core integration, and multi-step task decomposition, the original LLM data format has not been updated to match current requirements. This causes:

1. **Context Loss**: LLM forgets why it called a tool after receiving results
2. **Tool Selection Inaccuracy**: Similar tools get confused, over-generalization to "generic" tools
3. **Format Conversion Overhead**: Excessive transformations between components
4. **Debugging Difficulty**: Hard to trace data flow across components

### Solution: ExecutionContext-Driven Architecture

Introduce a semantic "backpack" (`ExecutionContext`) that travels through the entire execution chain, ensuring LLM always understands:
- **What** it's doing
- **Why** it's doing it
- **What** it has learned so far

## Part 1: ExecutionContext Core Structure

### Core Concept

`ExecutionContext` is not just parameter passing - it carries **all semantic information** needed for LLM to understand the task.

### Structure Definition

```rust
/// Execution context - persists throughout task lifecycle
pub struct ExecutionContext {
    /// Original user intent (immutable)
    pub original_intent: UserIntent,

    /// Current goal (may refine as task decomposes)
    pub current_goal: Goal,

    /// Decision trail (why these choices were made)
    pub decision_trail: Vec<Decision>,

    /// Acquired knowledge (valuable results from tool calls)
    pub acquired_knowledge: Vec<Knowledge>,

    /// Current execution phase
    pub phase: ExecutionPhase,
}

/// User intent - preserves raw input + structured understanding
pub struct UserIntent {
    /// Raw user input
    pub raw_input: String,           // "Help me deploy this project to the server"

    /// Structured interpretation
    pub understood_as: String,       // "Deploy current project to remote server"

    /// Key entities extracted
    pub key_entities: Vec<Entity>,   // [project, server]

    /// Implicit expectations
    pub implicit_expectations: Vec<String>, // ["Don't break existing service", "Verify deployment success"]
}

/// Current goal
pub struct Goal {
    /// Goal description
    pub description: String,         // "Execute deployment script"

    /// Success criteria
    pub success_criteria: String,    // "Script returns 0 and service is accessible"

    /// Link to parent goal
    pub parent_goal: Option<String>,
}

/// Decision record
pub struct Decision {
    /// What was decided
    pub choice: String,

    /// Why this choice was made
    pub reasoning: String,

    /// Alternatives considered
    pub alternatives: Vec<String>,

    /// Timestamp
    pub timestamp: i64,
}

/// Execution phase
pub enum ExecutionPhase {
    /// Understanding user intent
    Understanding,
    /// Planning execution steps
    Planning,
    /// Executing tools
    Executing,
    /// Validating results
    Validating,
    /// Summarizing for user
    Summarizing,
}
```

### Design Principles

1. **`original_intent` is immutable** - Regardless of task decomposition, always preserve user's original intent
2. **`decision_trail` records "why"** - Not just what was done, but reasoning behind choices
3. **`acquired_knowledge` is filtered** - Not all tool outputs, only information valuable to the task

## Part 2: Enhanced ToolCall Data Flow

### Problem

Current flow:
```
ToolCallInfo { id, name, arguments }  →  Execute  →  ToolCallResult { content }
```
LLM only sees "what the result is", not "why it was called" or "what this result means".

### Enhanced Structures

```rust
/// Enhanced tool call information
pub struct ToolCallInfo {
    pub id: String,
    pub name: String,
    pub arguments: Value,

    // === New fields ===
    /// Purpose of this call (LLM generated)
    pub purpose: String,           // "Find project config files to determine database connection method"

    /// Expected outcome type
    pub expected_outcome: String,  // "List of file paths containing database configuration"

    /// Relation to current goal
    pub goal_relation: GoalRelation,
}

/// How this tool call relates to the goal
pub enum GoalRelation {
    /// Directly achieves the goal
    DirectlyAchieves,
    /// Gathers information for subsequent decisions
    GathersInformation,
    /// Validates previous results
    Validates,
    /// Prepares for subsequent steps
    Prepares,
}

/// Enhanced tool call result
pub struct ToolCallResult {
    pub tool_call_id: String,
    pub name: String,
    pub content: String,
    pub success: bool,

    // === New fields ===
    /// Result summary (generated by system or LLM)
    pub summary: Option<String>,   // "Found 3 config files: config.toml, .env, settings.json"

    /// Contribution to goal
    pub goal_contribution: String, // "Config file locations confirmed, can now read specific configs"

    /// Extracted key information (auto-added to acquired_knowledge)
    pub extracted_knowledge: Vec<Knowledge>,
}

/// Knowledge fragment
pub struct Knowledge {
    /// Knowledge key
    pub key: String,      // "database_config_path"

    /// Knowledge value
    pub value: String,    // "./config/database.toml"

    /// Source of this knowledge
    pub source: String,   // "search_files tool call"

    /// Confidence level
    pub confidence: f32,  // 0.95
}
```

### Data Flow Change

```
Before call: LLM must declare purpose + expected_outcome
     ↓
After execution: System auto-generates summary + goal_contribution
     ↓
Return: LLM sees complete "why → result → meaning" chain
```

## Part 3: Structured Tool Description Format

### Problem

Current tool descriptions are free-form text, making it hard for LLM to distinguish similar tools:
```rust
// Current
description: "Search for files in the project"  // Too generic
```

### Enhanced Structure

```rust
/// Enhanced tool definition
pub struct ToolDefinition {
    pub name: String,
    pub display_name: String,

    // === Structured description ===
    /// One-line summary (for LLM quick filtering)
    pub summary: String,

    /// Core capabilities (precise enumeration)
    pub capabilities: Vec<Capability>,

    /// Explicitly unsuitable scenarios (prevent misuse)
    pub not_suitable_for: Vec<String>,

    /// Differentiation from similar tools
    pub differentiation: Vec<ToolDiff>,

    /// Typical use cases (positive examples)
    pub use_when: Vec<String>,

    /// Parameter schema
    pub parameters: Value,
}

/// Capability description
pub struct Capability {
    /// Action verb
    pub action: String,      // "search"

    /// Target of action
    pub target: String,      // "file names and paths"

    /// Scope limitation
    pub scope: String,       // "current project directory"

    /// Output type
    pub output: String,      // "list of full paths for matching files"
}

/// Tool differentiation
pub struct ToolDiff {
    /// The other tool being compared
    pub other_tool: String,  // "search_content"

    /// What this tool does
    pub this_tool: String,   // "searches by file name/path pattern"

    /// What the other tool does
    pub other_is: String,    // "searches by file content keywords"

    /// When to choose this tool
    pub choose_this_when: String,  // "know file name but not location"

    /// When to choose the other tool
    pub choose_other_when: String, // "know content characteristics but not file name"
}
```

### Practical Example

```rust
ToolDefinition {
    name: "search_files",
    summary: "Find file locations by name or path pattern",

    capabilities: vec![
        Capability {
            action: "search",
            target: "file names, extensions, path patterns",
            scope: "project directory tree",
            output: "full paths of matching files",
        }
    ],

    not_suitable_for: vec![
        "searching text content inside files",
        "reading or modifying files",
        "searching files outside project directory",
    ],

    differentiation: vec![
        ToolDiff {
            other_tool: "search_content",
            this_tool: "matches file name/path",
            other_is: "matches file content",
            choose_this_when: "know what the file is called or which directory it's in",
            choose_other_when: "know what content the file contains",
        },
        ToolDiff {
            other_tool: "read_file",
            this_tool: "finds file location",
            other_is: "reads content of known file",
            choose_this_when: "unsure where file is",
            choose_other_when: "already know full path and need content",
        }
    ],

    use_when: vec![
        "user mentions specific file name like 'config.toml'",
        "need to find certain file types like 'all .rs files'",
        "unsure where config file is located in project",
    ],
}
```

## Part 4: LLM Message Format Optimization

### Core Problem

Data structures are designed, but ultimately need to be converted to text format LLM can understand. Key is ensuring LLM **sees necessary context with each request** while controlling token consumption.

### System Prompt Structured Template

```rust
/// Generate system prompt for LLM
pub struct SystemPromptBuilder {
    /// Base persona (fixed)
    base_persona: String,

    /// Current execution context (dynamically injected)
    execution_context: Option<ExecutionContext>,

    /// Available tools (structured format)
    available_tools: Vec<ToolDefinition>,
}

impl SystemPromptBuilder {
    pub fn build(&self) -> String {
        format!(r#"
{base_persona}

## Current Task Context
{context_section}

## Available Tools
{tools_section}

## Tool Call Specification
When calling tools, you must declare:
- purpose: Why you are calling this tool
- expected_outcome: What result you expect to get
"#)
    }
}
```

### Context Injection Format

```markdown
## Current Task Context

**User Original Intent**: Help me deploy this project to the server
**Understood As**: Compile and deploy current Rust project to remote server
**Implicit Expectations**: Don't break existing service, verify deployment success

**Current Goal**: Find project build configuration
**Success Criteria**: Find Cargo.toml and confirm build target

**Acquired Information**:
- project_type: Rust (source: directory scan, confidence: 0.95)
- has_docker: true (source: file detection, confidence: 1.0)

**Decision History**:
1. Chose to analyze project structure first, need to determine build method
2. Detected Dockerfile, may use container deployment
```

### Tool Result Return Format

```markdown
## Tool Execution Result

**Tool**: search_files
**Call Purpose**: Find project config files to determine build method
**Expected Result**: List of files containing build configuration

**Execution Result**: Success
**Result Summary**: Found 3 config files
**Detailed Output**:
- ./Cargo.toml
- ./Dockerfile
- ./deploy.sh

**Contribution to Goal**: Confirmed project is Rust, has Docker support and deployment script
**Extracted Key Information**:
- build_config: ./Cargo.toml
- deploy_method: docker + script

**Next Step Suggestion**: Read Cargo.toml to confirm build parameters, or directly execute deploy.sh
```

### Token Optimization Strategy

```rust
/// Context verbosity levels
pub enum ContextVerbosity {
    /// First request: full context
    Full,
    /// Subsequent requests: incremental + key references only
    Incremental,
    /// Token-constrained: only core information
    Minimal,
}

impl ExecutionContext {
    /// Generate context string based on verbosity level
    pub fn to_prompt(&self, verbosity: ContextVerbosity) -> String {
        match verbosity {
            ContextVerbosity::Full => self.full_context_string(),
            ContextVerbosity::Incremental => self.incremental_context_string(),
            ContextVerbosity::Minimal => self.minimal_context_string(),
        }
    }

    fn minimal_context_string(&self) -> String {
        format!(
            "Goal: {}\nKey info: {:?}",
            self.current_goal.description,
            self.acquired_knowledge.iter()
                .filter(|k| k.confidence > 0.8)
                .map(|k| format!("{}={}", k.key, k.value))
                .collect::<Vec<_>>()
        )
    }
}
```

## Part 5: Migration Strategy

### Phase 1: Add New Structures (Non-breaking)

1. Add `ExecutionContext` to `core/src/components/types.rs`
2. Add `purpose`, `expected_outcome` fields to `ToolCallInfo` (optional)
3. Add `summary`, `goal_contribution` fields to `ToolCallResult` (optional)

### Phase 2: Enhance Tool Definitions

1. Create `ToolDefinitionV2` with structured fields
2. Migrate existing tool descriptions one by one
3. Add `differentiation` for similar tool pairs

### Phase 3: Update Message Generation

1. Implement `SystemPromptBuilder`
2. Implement `ContextVerbosity` logic
3. Update `ConversationHistory` to include context

### Phase 4: Integration Testing

1. Test context retention across multi-step tasks
2. Test tool selection accuracy with similar tools
3. Benchmark token consumption changes
