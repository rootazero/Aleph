# PII Filtering Design ŌĆö Gateway-Level Privacy Protection

> **Date**: 2026-02-23
> **Status**: Approved
> **Scope**: Real-time PII filtering on outbound LLM API calls

---

## 1. Problem

When Aleph sends messages to LLM providers (Anthropic, OpenAI, etc.), the payload may contain personally identifiable information (PII) ŌĆö phone numbers, ID card numbers, bank card numbers, emails, API keys, etc. This information travels over HTTPS to third-party servers with no filtering.

**Current state in Aleph:**
- `utils/pii.rs` has `scrub_pii()` ŌĆö used only for log scrubbing (accepts false positives)
- `logging/pii_filter.rs` ŌĆö tracing layer for log output
- `secrets/` ŌĆö encrypted vault for API keys (not for message-level PII)
- **No filtering exists on the Provider ŌåÆ API data path**

## 2. Goal

Add a precision PII filtering engine to the Provider layer. Filter outbound messages before they reach LLM APIs. Minimize false positives (which degrade LLM comprehension) while catching structured PII.

**Non-goals (this phase):**
- Bidirectional filtering (no response-side restoration)
- Auto-learning / entity watchlist (deferred to Phase 2)
- NER-based detection for unstructured PII (names, addresses)
- Historical session scanning / audit

## 3. Architecture

### 3.1 Insertion Point: HttpProvider::execute()

All LLM API calls flow through `HttpProvider::execute()` which builds a `RequestPayload` and sends it via `ProtocolAdapter`. The PII filter sits between payload construction and HTTP dispatch:

```
Thinker.call_llm_with_level()
  ŌåÆ provider.process(input, system_prompt)
    ŌåÆ HttpProvider.execute(RequestPayload)
      ŌåÆ **PiiEngine::filter(payload.input)**    ŌåÉ insertion point
      ŌåÆ adapter.build_request(filtered_payload)
      ŌåÆ request.send()
```

**Why here:**
- Closest to the exit ŌĆö nothing can bypass it
- Covers all protocols (Anthropic, OpenAI, Gemini, Ollama, etc.)
- Single change point ŌĆö no per-provider modifications needed
- System prompt is NOT filtered (generated by Aleph, no user PII)

### 3.2 Configuration

```toml
[privacy]
# Global toggle
pii_filtering = true

# Per-severity behavior: "block" | "warn" | "off"
# "block" = replace with placeholder before sending
# "warn"  = log detection but send unmodified
# "off"   = skip detection entirely
id_card = "block"         # Critical
bank_card = "block"       # High
phone = "block"           # High
api_key = "block"         # Critical
ssh_key = "block"         # Critical
email = "warn"            # Medium
ip_address = "off"        # Low

# Skip filtering for local/self-hosted providers
exclude_providers = ["ollama"]
```

Supports hot-reload via existing `config/reload.rs` mechanism.

### 3.3 Audit Logging

When PII is detected:

```rust
tracing::warn!(
    rule = match.rule_name,
    severity = %match.severity,
    action = "blocked",       // or "warned"
    provider = provider_name,
    "PII detected and filtered before API call"
);
```

No raw PII values in logs. Only rule name, severity, and action.

### 4.2 Core Abstractions

```rust
/// A single PII detection rule
pub trait PiiRule: Send + Sync {
    fn name(&self) -> &str;
    fn severity(&self) -> PiiSeverity;
    fn placeholder(&self) -> &str;           // e.g., "[PHONE]", "[ID_CARD]"
    fn detect(&self, text: &str) -> Vec<PiiMatch>;
}

pub struct PiiMatch {
    pub rule_name: String,
    pub start: usize,
    pub end: usize,
    pub matched_text: String,
    pub severity: PiiSeverity,
    pub confidence: f32,                     // 0.0-1.0
}

pub enum PiiSeverity { Critical, High, Medium, Low }

pub enum PiiAction { Block, Warn, Off }

/// Main engine ŌĆö thread-safe, globally accessible
pub struct PiiEngine {
    rules: Vec<Box<dyn PiiRule>>,
    allowlist: PiiAllowlist,
    config: PrivacyConfig,
}
```

### 4.3 Key Design: PiiEngine::filter()

```rust
impl PiiEngine {
    /// Filter PII from text, returning sanitized version
    pub fn filter(&self, text: &str) -> FilterResult {
        if !self.config.pii_filtering {
            return FilterResult::unchanged(text);
        }

        let mut detections: Vec<PiiMatch> = Vec::new();

        // Run all rules
        for rule in &self.rules {
            let action = self.config.action_for(rule.severity());
            if action == PiiAction::Off {
                continue;
            }

            let matches = rule.detect(text);

            // Filter through allowlist
            let filtered = matches.into_iter()
                .filter(|m| !self.allowlist.is_allowed(&m.matched_text, rule.name()))
                .collect::<Vec<_>>();

            detections.extend(filtered);
        }

        // Sort by position (reverse) for safe replacement
        detections.sort_by(|a, b| b.start.cmp(&a.start));

        // Apply replacements (block) or just record (warn)
        let mut result = text.to_string();
        let mut blocked = Vec::new();
        let mut warned = Vec::new();

        for detection in &detections {
            let action = self.config.action_for(detection.severity);
            match action {
                PiiAction::Block => {
                    result.replace_range(
                        detection.start..detection.end,
                        &detection.placeholder(),
                    );
                    blocked.push(detection.clone());
                }
                PiiAction::Warn => {
                    warned.push(detection.clone());
                }
                PiiAction::Off => {}
            }
        }

        FilterResult { text: result, blocked, warned }
    }
}
```

## 5. Anti-False-Positive Rules

These rules are critical for gateway-level filtering where false positives degrade LLM comprehension.

### 5.1 Chinese Mobile Phone (`phone.rs`)

**Base pattern**: `1[3-9]\d{9}` with word boundary

**Anti-false-positive checks:**
1. **Hex boundary**: Skip if preceded/followed by `[a-fA-F]` (UUID fragment)
2. **Timestamp context**: Skip if surrounding text contains `timestamp`, `time`, `date`, `created_at`, `updated_at`
3. **Test number exclusion**: Skip `13800138000`, `18888888888`, etc.

### 5.2 Chinese ID Card (`id_card.rs`)

**Base pattern**: `\d{17}[\dXx]` with word boundary

**Structural validation (all must pass):**
1. **Region code**: First 2 digits must be 11-82 (valid province codes)
2. **Date validation**: Digits 7-14 must form valid YYYYMMDD (1900-2100, valid month/day)
3. **Checksum**: ISO 7064 MOD 11-2 algorithm on all 18 digits

This eliminates Discord Snowflake IDs, random 18-digit numbers, and most false positives.

### 5.3 Bank Card (`bank_card.rs`)

**Base pattern**: `\d{16,19}` with word boundary

**Anti-false-positive checks:**
1. **Decimal point exclusion**: Skip if preceded by `.` (JSON float)
2. **Luhn checksum**: Must pass Luhn algorithm
3. **Known BIN prefix**: Check first 4-6 digits against known bank prefixes (62xxxx for UnionPay, 4xxx for Visa, etc.)

### 5.4 Email (`email.rs`)

**Base pattern**: Standard RFC 5322 simplified

**Exclusions:**
- `noreply@*`, `no-reply@*`, `donotreply@*`
- `*@example.com`, `*@test.com`, `*@localhost`
- Domains containing: `example`, `test`, `demo`, `sample`, `mock`

### 5.5 API Key (`api_key.rs`)

**Prefix-only matching** (no generic long-string detection):
- `sk-` (OpenAI)
- `sk-ant-` (Anthropic)
- `ghp_` (GitHub)
- `AKIA` (AWS)
- `xox` (Slack)
- `tvly-` (Tavily)
- `Bearer ` + 20+ chars (OAuth/JWT)
- `-----BEGIN` (SSH/PGP keys)

### 5.6 IP Address (`ip_address.rs`)

**Pattern**: Standard IPv4 dotted decimal

**Exclusions:**
- `127.0.0.1`, `0.0.0.0`
- `192.168.*.*`, `10.*.*.*`, `172.16-31.*.*` (RFC 1918)
- `localhost`, `*.local`, `*.internal`

## 9. Testing Strategy

### Unit Tests (per rule)
- **Positive**: Real PII samples correctly detected
- **Negative**: All 6 anti-false-positive scenarios
  - Discord Snowflake IDs ŌåÆ NOT matched as ID card
  - UUID fragments ŌåÆ NOT matched as phone
  - Unix timestamps ŌåÆ NOT matched as phone
  - JSON floats ŌåÆ NOT matched as bank card
  - URL slugs ŌåÆ NOT matched as API key
  - Random 18-digit numbers ŌåÆ NOT matched as ID card (checksum fails)
- **Allowlist**: Test numbers, system emails correctly excluded

### Integration Tests
- Mock HttpProvider ŌåÆ verify PII filtered before "sending"
- Config-driven: verify `exclude_providers` skips filtering
- Action modes: verify `block` replaces, `warn` logs only, `off` skips

### Benchmarks
- Filter latency < 1ms for typical message sizes (< 10KB)
- Regex compilation < 10ms (one-time at startup)

## 10. Future Evolution

| Phase | Scope | Description |
|-------|-------|-------------|
| **Phase 1 (this)** | Real-time regex | 7 rule types + anti-false-positive + allowlist |
| **Phase 2** | Auto-learning | Scan historical sessions ŌåÆ build entity watchlist ŌåÆ auto-intercept |
| **Phase 3** | Local NER | GLiNER or fine-tuned BERT for names, addresses, companies |

Phase 1 covers ~90% of structured PII leaks. Phase 2-3 are independent additions that don't require architectural changes.
